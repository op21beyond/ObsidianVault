# NPU RTL 검증 자동화 에이전트 구현 계획

## 1. 배경 조사

### 1.1 LLM 기반 하드웨어 검증 연구 동향

최근 몇 년 사이 대형 언어 모델(LLM)을 EDA(전자 설계 자동화)에 적용하려는 연구가 빠르게 진행되고 있다. **AssertLLM**는 건축사양 문서에 포함된 자연어 문장과 파형 그림을 구조화된 표현으로 변환한 뒤 LLM을 이용해 SystemVerilog Assertion(SVA)를 자동 생성한다. 논문에 따르면 AssertLLM은 GPT‑4o, GPT‑3.5 등 범용 모델보다 더 정확하고 높은 품질의 어설션을 생성할 수 있다는 평가를 받는다. 이는 LLM을 활용한 사양 문서 이해와 검증 구성요소 생성이 가능하다는 근거를 제공한다.

범용 검증 환경인 UVM(Testbench)에 LLM을 접목한 연구도 진행 중이다. **UVM2** 프레임워크는 LLM을 이용하여 UVM 구성요소(시퀀서, 드라이버 등)와 테스트 케이스를 자동으로 생성한 후, 커버리지 결과를 분석하여 테스트를 반복 개선한다. 이 연구에서 UVM2는 경험 많은 엔지니어보다 최대 38.82배 빠르게 테스트벤치를 구성하고 기존 방법 대비 20% 이상 높은 코드/기능 커버리지를 달성하였다. 이는 LLM이 테스트벤치 생성과 커버리지 기반 반복 개선에 효과적임을 보여 준다.

또한, **TB or Not TB**는 Verilog 테스트 자극을 자동 생성하기 위해 커버리지 기반 직접 선호 최적화(Coverage‑Driven Direct Preference Optimization, CD‑DPO)를 적용하였다. 논문에서는 파이썬 기반 시뮬레이션에서 높은 커버리지를 기록한 테스트벤치를 우수 사례로 선택하고 모델을 미세조정하여 LLM이 커버리지 신호를 학습하도록 한다. 이 방법은 기존 오픈 소스 및 상용 도구를 능가하는 코드 커버리지 향상을 달성하였다. 특히 LLM을 이용한 테스트벤치 생성이 무작위적 자극 생성보다 월등히 효율적임을 강조한다.

학계에서는 RAG(검색 증강 생성)를 활용하여 사양 문서, C 모델, 설계 코드 등 도메인 지식을 벡터 데이터베이스에 저장하고 LLM의 프롬프트로 주입해 응답 품질을 높이는 일반적인 아키텍처를 제안한다. LLM 응용 아키텍처는 세 단계(전처리–LLM 질의–후처리)로 구성되며, 전처리 단계에서 큰 문서를 작은 청크로 나누어 벡터 임베딩을 생성하고, LLM 질의 단계에서 벡터 검색 결과를 포함한 프롬프트를 사용해 정확도를 향상시킨다.

### 1.2 커버리지 기반 검증 및 버그 탐지

전통적인 RTL 검증은 기능 커버리지와 코드 커버리지를 통해 테스트 품질을 측정한다. UVM과 같은 프레임워크에서는 시퀀서·드라이버·모니터 등 계층형 컴포넌트를 사용하여 자극을 생성하고 DUT(Design Under Test)와 비교 모델을 연결하는 스코어보드를 구성한다. 논문에서는 기능 검증 단계가 IC 프런트엔드 설계 시간의 70% 이상을 차지하며, 테스트벤치 생성과 테스트케이스 보완이 가장 시간 소모적인 부분이라고 밝힌다. 이러한 커버리지 지향 기반의 검증 흐름을 LLM과 결합하면 자동화된 테스트 생성과 반복 개선이 가능하다.

## 2. 요구사항 분석 및 설계 원칙

### 2.1 요구사항 정리

사용자의 요구는 다음과 같다:

1. **NPU RTL 검증 자동화** – NPU 명령어 아키텍처 사양, 사용자 가이드, 레퍼런스 시뮬레이터 C 코드를 입력으로 사용하여 DUT RTL의 검증 코드를 자동으로 생성해야 한다.
    
2. **LLM 기반 하드웨어 설계 검증** – 명령어 사양 해석, 테스트 벡터 생성, 검증 assertions 생성 등을 LLM을 활용해 자동화한다.
    
3. **커버리지 기반 테스트 코드 생성 및 반복 개선** – 테스트 실행 후 코드/기능 커버리지 분석을 통해 부족한 부분을 LLM이 인식하고 테스트를 추가한다.
    
4. **테스트 실행 및 버그 검출** – Verilog RTL과 C reference simulator를 실행하여 결과를 비교하고 불일치나 assertion 위반을 버그로 기록한다.
    

### 2.2 설계 원칙

- **유연성** – NPU 사양 변화나 다른 RTL 블록에도 적용할 수 있도록 모듈화된 구조로 설계한다.
    
- **확장성** – 대규모 명령어 세트나 여러 핵심 모듈을 처리할 수 있도록 데이터 저장과 테스트 생성 파이프라인을 확장 가능하게 만든다.
    
- **신뢰성** – LLM 응답의 신뢰도를 높이기 위해 RAG 방식을 사용하고, 테스트 결과와 커버리지 데이터를 통해 자동 생성된 코드를 검증한다.
    
- **반복적 개선** – 커버리지 피드백을 LLM에 제공하여 미달 커버리지를 채우도록 테스트 케이스를 자동 확장한다.
    

## 3. 전체 아키텍처 설계

아래 그림은 제안하는 NPU RTL 검증 자동화 에이전트의 전체 아키텍처이다. 파일에는 문서 처리, LLM 기반 테스트 생성, 시뮬레이션 및 커버리지 분석 모듈이 포함되어 있다.

### 3.1 아키텍처 다이어그램

설명:

1. **지식 수집 및 전처리** – 사양 문서, 사용자 가이드, C 레퍼런스 시뮬레이터를 수집하여 텍스트와 함수 정보를 추출한다. PDF에서 파라그래프와 테이블을 분할하고, 코드 파일에서 함수 시그니처와 주석을 추출해 데이터 청크를 만든다. 각 청크는 임베딩 모델을 통해 벡터 데이터베이스에 저장된다.
    
2. **검증 목표 식별** – 사용자로부터 검증하려는 NPU 버전과 명령어 범위를 입력받는다. 요구사항 분석 모듈은 명령어 목록을 파싱하여 각 명령어의 동작, 피연산자, 메모리 접근 패턴 등을 정의한다.
    
3. **테스트벤치/스코어보드 생성** – LLM은 벡터 DB로부터 각 명령어의 사양과 C reference 구현을 검색(RAG)하여 Verilog 테스트벤치 요소를 생성한다. UVM 환경의 세부 구성을 생략하고 Cocotb 기반 테스트벤치를 사용하여 파이썬으로 시뮬레이션과 결과 비교를 간소화할 수 있다.
    
    - **Stimulus Generator** – LLM은 명령어 형식과 제약 조건을 고려하여 다양한 입력 시나리오와 코너 케이스를 생성한다. UVM2 사례처럼 커버리지 피드백을 통해 반복적으로 개선된다.
        
    - **Scoreboard & Reference Model** – C reference 코드를 Python extension 형태로 호출하여 시뮬레이션 결과를 생성하고, RTL 시뮬레이션 결과와 비교한다.
        
4. **시뮬레이션 및 커버리지 분석** – 생성된 테스트는 Verilator 또는 Icarus Verilog와 같은 시뮬레이터로 실행한다. 실행 중에는 SystemVerilog assert뿐만 아니라 LLM이 생성한 SVA assertions도 포함한다. 코드 커버리지와 기능 커버리지 리포트를 생성하여 벡터 DB와 LLM에 제공한다.
    
5. **버그 검출 및 리포트** – 스코어보드는 RTL 출력과 레퍼런스 시뮬레이터 출력을 비교하여 버그를 식별한다. 버그가 발생한 테스트의 인풋, 출력, 에러 메시지, 커버리지 정보를 JSON 또는 데이터베이스에 저장하고, 사용자에게 요약 리포트를 제공한다.
    
6. **커버리지 기반 피드백 루프** – 커버리지 보고서에서 미달 커버리지를 파악하여 LLM에게 전달한다. LLM은 새로운 테스트를 생성하고 업데이트된 테스트벤치를 다시 시뮬레이션한다. 필요 시 불충분한 SVA 어설션을 추가 생성한다. ‘TB or Not TB’ 연구에서와 같이 커버리지 신호를 최적화 목표로 삼아 모델 미세조정을 적용하는 것도 가능하다.
    

## 4. 주요 구성 요소 및 구현 세부안

### 4.1 문서 처리 및 지식 그래프 구축

- **파서 모듈**: PDF 파서 및 C 소스 파서를 작성하여 사양 문서와 사용자 가이드에서 명령어 설명, 파라미터, 상태 머신 동작을 추출한다. 자연어 문장은 문장 단위로 분리하고, 테이블은 구조화된 JSON으로 변환한다.
    
- **임베딩 및 벡터 데이터베이스**: 추출한 청크에 대하여 임베딩 모델(e.g. open‑source embedding model)을 사용하여 벡터를 생성하고 FAISS 또는 Chroma와 같은 벡터 DB에 저장한다.
    
- **메타데이터 저장**: 각 청크에는 문서 출처와 라인 번호, 명령어 이름 등을 메타데이터로 저장하여 추후 인용과 링크 제공이 가능하게 한다.
    

### 4.2 LLM 기반 테스트벤치 및 assertion 생성

- **RAG 기반 프롬프트**: LLM 호출 전에 명령어 이름, 관련 청크, C 참조 코드 조각 등을 벡터 DB에서 검색하여 프롬프트에 포함한다. 이는 hallucination을 줄이고 사양에 부합하는 테스트를 생성하는 데 필수적이다.
    
- **테스트 케이스 생성**: 각 명령어에 대해 LLM에게 입력 값의 범위, 경계 조건, 메모리 충돌 시나리오 등을 포함한 다양한 시퀀스 생성을 요청한다.
    
- **assertion 생성**: AssertLLM 방식처럼 LLM에게 상태 머신이나 타이밍 파형 설명을 전달하여 SystemVerilog Assertion(SVA) 형식으로 변환하도록 한다. 버그 검출과 디버깅에 도움을 주는 assertion을 자동으로 포함한다.
    
- **코드 안전성 확인**: 생성된 Verilog/C++ 테스트 코드에 대해 기본적인 정적 분석(문법 검사, 시뮬레이터 구문 검사)을 수행하여 실행 오류를 최소화한다.
    

### 4.3 테스트 실행 및 스코어보드

- **시뮬레이터 통합**: 무료 시뮬레이터(Verilator, Icarus Verilog)를 사용하여 RTL을 컴파일하고 Cocotb를 통해 Python 드라이버에서 제어한다. Verilator는 C++로 생성된 시뮬레이션 모델을 통해 고속 실행이 가능하며, 커버리지 집계를 위한 VCD(Vital Change Dump) 파일을 제공한다.
    
- **레퍼런스 모델**: C 시뮬레이터를 Python에서 호출할 수 있도록 FFI(파이썬 ctypes 또는 CFFI) 래퍼를 작성한다. 각 테스트 입력에 대해 C 모델을 실행하고 RTL 시뮬레이터 출력과 비교하여 정답 여부를 판단한다.
    
- **스코어보드**: 테스트 케이스 식별자, 입력 데이터, DUT 출력, C 모델 출력, assertion 실패 여부, 커버리지 결과를 기록하는 스코어보드 모듈을 작성한다. 결과는 JSON/CSV로 기록하여 추후 분석 및 학습 데이터로 사용한다.
    

### 4.4 커버리지 분석 및 피드백

- **코드 커버리지**: 시뮬레이터가 생성한 VCD 또는 LCOV 파일을 분석하여 각 RTL 라인의 실행 여부를 확인한다.
    
- **기능 커버리지**: SVA 어설션 및 covergroup을 통해 명령어 조합, 파이프라인 상태 등 사용자 정의 포인트에 대한 커버리지를 수집한다.
    
- **피드백 루프**: 미달 커버리지 항목을 파싱하여 LLM 프롬프트에 “아직 실행되지 않은 명령어 조합 X, Y를 테스트하십시오”와 같은 형태로 전달한다. TB or Not TB 연구에서 제안한 CD‑DPO 방법처럼 높은 커버리지를 얻은 테스트와 낮은 커버리지를 얻은 테스트를 비교 데이터로 삼아 LLM 파인튜닝이나 강화학습을 적용할 수 있다.
    

### 4.5 버그 보고 및 사용자 인터페이스

- **버그 리포트**: 테스트 중 발견된 assertion 위반, 레퍼런스 모델과 불일치, 시뮬레이션 에러 등을 수집하여 HTML/Markdown 리포트로 제공한다. 버그 발생 시 입력 파라미터, 해당 명령어, 타임스탬프를 명시하여 재현을 용이하게 한다.
    
- **대시보드**: 커버리지 통계와 테스트 진행 상황을 시각화하는 대시보드를 제공한다. 사용자는 특정 명령어 또는 모듈의 검증 상태를 쉽게 파악할 수 있다.
    
- **설정 관리**: LLM 파라미터(temperature, max_tokens), 테스트 생성 개수, 커버리지 목표 등을 사용자가 설정할 수 있도록 한다.
    

## 5. 추진 전략

1. **프로토타입 개발** – Python 기반으로 문서 파서, 벡터 DB, LLM 호출, 시뮬레이션 자동화를 결합한 최소 기능 제품(MVP)을 만든다. 초기에는 일반적인 언어 모델을 사용하고 커버리지 피드백은 간단한 heuristic(“커버되지 않은 명령어 우선”)으로 처리한다.
    
2. **커버리지 기반 학습 개선** – TB or Not TB 연구처럼 커버리지 신호를 학습 데이터로 활용하여 LLM을 미세 조정한다. 높은 커버리지를 얻은 테스트케이스를 양성 사례로, 낮은 커버리지를 얻은 사례를 음성 사례로 사용해 preference-based training을 적용하면 테스트 생성 품질을 지속적으로 향상시킬 수 있다.
    
3. **assertion 자동 생성 고도화** – AssertLLM을 참고하여 파형 그림과 상태 머신 설명으로부터 assertions를 자동 추출하는 모듈을 도입한다. 이를 통해 버그 탐지 능력을 강화한다.
    
4. **확장 및 최적화** – 프로토타입 성능을 평가한 뒤, 필요에 따라 GPU 가속 시뮬레이터나 상용 EDA 도구와 연동하여 대규모 디자인에 적용한다. UVM2 연구에서처럼 설계 복잡도가 증가해도 실용적인 성능을 달성하도록 개선한다.
    

## 6. 결론

NPU와 같은 복잡한 RTL 설계의 검증에서는 광범위한 테스트와 치밀한 커버리지 분석이 요구된다. 최신 연구들은 LLM을 활용한 자동 테스트 생성과 UVM 환경의 반복 개선을 통해 검증 효율을 획기적으로 향상시키고 있음을 보고한다. 본 구현 계획은 이러한 연구 결과를 바탕으로, 사양 문서와 레퍼런스 모델을 RAG 방식으로 활용하여 테스트벤치와 assertions를 자동 생성하고, 커버리지 기반 피드백 루프를 통해 테스트를 지속적으로 개선하는 완전 자동화된 검증 에이전트를 제시하였다.

이러한 시스템을 구축함으로써 NPU 설계 검증에 필요한 반복적인 수작업을 크게 줄이고, 설계 품질을 향상시키며, 새로운 명령어나 기능이 추가될 때 빠르게 대응할 수 있을 것이다.
